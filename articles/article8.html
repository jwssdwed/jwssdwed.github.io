Java Virtual Machine

Mar 23, 2018 / Written by Chengyu

Day 1:

Java Memory Usage:

Unlike C, C++, in which developers are required to manage the memory by themselves, Java developers don’t need to be bothered with those complicated issues. One the one hand, it is more efficient and easier to develop, on the other hand, developer may not be able to detect and solve problem productively when a memory problem occurs. Thus, in order to debug the memory issue in the future life, it is extremely important for a Java developer to be familiar with how a JVM manage memory.
When a JVM is executing a Java program, it will divide the memory into various separated areas, which have their own corresponding creating, ending time.

  • Program Counter Register
    • This takes a small memory area which is used to locate the executing codes, and indicate which line of code the machine is in currently. Every thread has its own personal Register.
  • Java Virtual Machine Stack
    • Every thread has its own VM Stack and the lifetime of them are same. Many developers prefer to separate the memory merely using heap and stack, however, this division is too narrow to Java memory areas. Usually, the stack has a local variable table/list in which all the primitive types and a reference to (an) object. Except for the long & double, all the type only takes one slot. The table is formed during compile, which means it will never change during executing/running time.
  • Native Method Stack
    • The effect of this is very similar to VM Stack, and the only difference is that this stack on1y supports the native method, which is written in any language except fro Java.
  • Java Heap:
    • For most application, Java heap is the most significant area among all the memories that Java manipulates. Each process has one corresponding head and all the threads under this head share it. The usage of this is store all the objects when they are created. Here is one definition from JDK: “The head is the JAVA Area from which memory for all class instances and arrays is allocated”. Java heap is also the critical area for GC (Garbage Collection).
  • Method Area:
    • Analogue to the Heap, all threads belonging to the same process share this area, and it used to store loaded classes, static variables, constants and compiled codes. Different from Heap, it can prevent the GC so that some variables defined inside can exist “Permanently”.
